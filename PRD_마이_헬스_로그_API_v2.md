# 마이 헬스 로그 API PRD

## 1. 문서 정보

- 프로젝트명: 마이 헬스 로그 API
- 제품 유형: 개인 건강 기록 관리 REST API
- 개발 환경: Python, FastAPI, SQLite, Docker
- 저장 방식: SQLite 관계형 데이터베이스
- 인증 방식: HTTP Basic Authentication
- 문서 목적: 서비스 요구사항, 데이터 정책, API 범위, 테스트 시나리오를 확정한다.

---

## 2. 제품 개요

마이 헬스 로그 API는 사용자가 체중, 키, 혈압, 공복혈당, 걸음 수, 수면 시간, 메모를 기록하고 조회·수정·삭제할 수 있도록 제공하는 개인 건강 기록 관리 REST API다.

입력된 건강 수치를 바탕으로 BMI를 계산하고, BMI·혈압·공복혈당 상태를 분류하며, 위험 수준에 따라 경고 메시지를 생성한다. 사용자는 자신의 기록만 조회하거나 변경할 수 있으며, 날짜 범위 검색과 전체 기록 통계 기능을 제공한다.

---

## 3. 해결하려는 문제

사용자의 건강 정보는 메모, 웨어러블 앱, 병원 기록 등 여러 곳에 흩어질 수 있다. 이 때문에 과거 수치를 찾기 어렵고, 장기 변화나 평균·최솟값·최댓값을 직접 계산해야 하는 불편이 발생한다.

본 프로젝트는 다음 문제를 해결한다.

- 건강 기록을 날짜별로 일관된 형식으로 저장하기 어렵다.
- 여러 사용자의 데이터를 안전하게 분리해야 한다.
- BMI와 건강 상태 분류를 매번 수동으로 계산해야 한다.
- 특정 기간의 기록을 찾기 어렵다.
- 장기간의 건강 수치를 통계로 확인하기 어렵다.
- 서버 재시작 이후에도 데이터가 유지되어야 한다.

## 7. 핵심 사용자 정책

### 7.1 사용자명

- 영문자, 숫자, 밑줄, 하이픈만 허용한다.
- 길이는 3자 이상 30자 이하로 제한한다.
- 저장 전 소문자로 정규화한다.
- 중복 사용자명은 허용하지 않는다.
- 중복 시 409 Conflict를 반환한다.

### 7.2 비밀번호

- 8자 이상 128자 이하로 제한한다.
- 평문 비밀번호를 저장하지 않는다.
- Argon2 해시값만 SQLite에 저장한다.

### 7.3 사용자별 데이터 분리

- 로그인한 사용자는 자신의 기록만 조회·수정·삭제할 수 있다.
- 다른 사용자의 기록 ID를 요청하더라도 404를 반환한다.
- 검색과 통계도 현재 사용자의 데이터만 사용한다.

### 7.4 하루 한 건 정책

- 한 사용자는 같은 날짜에 건강 기록을 한 건만 등록할 수 있다.
- `UNIQUE(user_id, date)` 제약조건을 사용한다.
- POST 요청에서 같은 날짜의 기록이 이미 존재하면 409 Conflict를 반환한다.
- PUT 요청에서 측정일을 변경할 때, 변경하려는 날짜에 현재 사용자의 다른 기록이 이미 존재하면 409 Conflict를 반환한다.
- 수정 대상 기록 자신의 기존 날짜는 충돌로 판단하지 않는다.
- 기존 날짜의 수치를 변경할 때는 PUT API를 사용한다.

## 8. 데이터 저장 정책

### 8.1 SQLite 선택 이유

본 프로젝트는 다중 사용자 관계, 사용자별 기록 소유권, 날짜 검색, 통계 처리, 데이터 무결성을 명확하게 관리해야 한다. 일정과 학습 범위를 고려하여 별도 서버 설치가 필요 없고 Python 표준 라이브러리로 사용할 수 있는 SQLite를 선택한다.

### 8.2 저장하는 원본 필드

건강 기록 테이블에는 다음 원본 값만 저장한다.

- date
- weight
- height
- systolic
- diastolic
- blood_sugar
- steps
- sleep_hours
- memo

### 8.3 계산 필드 저장 여부

다음 필드는 데이터베이스에 저장하지 않고 API 응답 시 계산한다.

- bmi
- bmi_category
- bp_category
- sugar_category
- warnings

이 값들은 원본 건강 수치에서 다시 계산할 수 있는 파생 데이터다. 데이터베이스에 중복 저장하면 향후 건강 분류 기준이 변경되었을 때 기존 데이터와 새 기준이 불일치할 수 있다. 따라서 원본 데이터만 저장하고, 조회 시 현재 분류 함수로 다시 계산한다.

### 8.4 날짜 저장 방식

- SQLite에는 ISO 8601 형식의 문자열 `YYYY-MM-DD`로 저장한다.
- API 입력과 응답에서는 Pydantic의 `datetime.date` 검증을 사용한다.

### 8.5 시간 정보

- 사용자와 건강 기록에 `created_at`을 저장한다.
- 건강 기록에는 `updated_at`도 저장한다.
- 시간은 UTC 기준 ISO 형식 문자열로 저장한다.
- `created_at`과 `updated_at`은 내부 데이터 관리와 향후 감사 이력 확장을 위한 필드다.
- 이번 버전의 `UserPublic` 및 `RecordResponse` API 응답에는 시간 정보를 노출하지 않는다.

## 9. 데이터베이스 구조

### 9.1 users

| 컬럼 | 타입 | 제약조건 | 설명 |
|---|---|---|---|
| id | INTEGER | PRIMARY KEY AUTOINCREMENT | 사용자 고유 ID |
| username | TEXT | UNIQUE, NOT NULL | 정규화된 사용자명 |
| hashed_password | TEXT | NOT NULL | Argon2 비밀번호 해시 |
| created_at | TEXT | NOT NULL | 가입 시각 |

### 9.2 health_records

| 컬럼 | 타입 | 제약조건 | 설명 |
|---|---|---|---|
| id | INTEGER | PRIMARY KEY AUTOINCREMENT | 기록 고유 ID |
| user_id | INTEGER | NOT NULL, FOREIGN KEY | 기록 소유 사용자 |
| date | TEXT | NOT NULL | 측정일 |
| weight | REAL | NOT NULL | 몸무게(kg) |
| height | REAL | NOT NULL | 키(cm) |
| systolic | INTEGER | NOT NULL | 수축기 혈압 |
| diastolic | INTEGER | NOT NULL | 이완기 혈압 |
| blood_sugar | INTEGER | NOT NULL | 공복혈당 |
| steps | INTEGER | NOT NULL, DEFAULT 0 | 하루 걸음 수 |
| sleep_hours | REAL | NOT NULL, DEFAULT 0 | 수면 시간 |
| memo | TEXT | NOT NULL, DEFAULT '' | 메모 |
| created_at | TEXT | NOT NULL | 생성 시각 |
| updated_at | TEXT | NOT NULL | 수정 시각 |

추가 제약조건:

```sql
UNIQUE(user_id, date)
FOREIGN KEY(user_id) REFERENCES users(id) ON DELETE CASCADE
```

관계:

```text
users 1 ───────── N health_records
```

## 10. 건강 상태 계산 규칙

### 10.1 BMI 계산

```text
BMI = 몸무게(kg) / 키(m)^2
```

소수점 둘째 자리까지 반올림한다.

### 10.2 BMI 분류

- 18.5 미만: 저체중
- 18.5 이상 23 미만: 정상
- 23 이상 25 미만: 과체중
- 25 이상: 비만

### 10.3 혈압 분류

- 수축기 140 이상 또는 이완기 90 이상: 고혈압
- 수축기 120 이상 또는 이완기 80 이상: 주의
- 그 외: 정상

### 10.4 공복혈당 분류

- 100 미만: 정상
- 100 이상 126 미만: 공복혈당장애
- 126 이상: 당뇨 의심

### 10.5 경고 생성

- BMI가 비만이면 BMI 경고를 추가한다.
- 혈압이 고혈압이면 혈압 경고를 추가한다.
- 공복혈당이 당뇨 의심이면 혈당 경고를 추가한다.

본 분류는 프로젝트 시연용 규칙이며 의료 진단을 대체하지 않는다.

## 12. 주요 오류 응답

| 상태 코드 | 발생 조건 |
|---|---|
| 400 | 검색 시작일이 종료일보다 늦음 |
| 401 | 인증 정보가 없거나 사용자명·비밀번호가 틀림 |
| 404 | 현재 사용자의 기록이 없거나 통계 대상이 없음 |
| 409 | 사용자명 중복, POST 동일 날짜 중복, PUT 날짜 변경 충돌 |
| 422 | 입력값 형식 또는 범위가 잘못됨 |

현재 로그인 사용자의 전체 기록을 대상으로 다음 값을 계산한다.

- 평균 몸무게

통계 대상 기록이 없으면 404를 반환한다.

## 14. 가상 데이터 시나리오

가상 데이터는 실제 인물을 나타내지 않는 시연용 합성 데이터다. 생활 습관과 건강 수치 사이의 인과관계를 입증하는 자료가 아니라, 장기 추세와 API 기능을 테스트하기 위한 가정 데이터다.

공통 기간:

```text
2025-07-01 ~ 2026-06-30
```

공통 수량:

```text
사용자 3명 × 365일 = 1,095건
```

### 14.1 demo_stable

꾸준히 운동하고 규칙적으로 생활하며 정상 범위의 건강 상태를 유지하는 사용자.

- 정상 BMI 범위에서 작은 변동
- 걸음 수는 비교적 높고 안정적
- 수면 시간은 대체로 충분
- 혈압과 공복혈당은 대부분 정상
- 피로·휴식일에 일시적인 변동 허용

### 14.2 demo_decline

초기에는 경고 단계이지만 운동 감소, 수면 부족, 체중 증가 등 잘못된 생활 습관으로 점차 악화되는 사용자.

- 초기에는 과체중·주의 경계
- 걸음 수와 수면 시간이 점진적으로 감소
- 몸무게가 완만하게 증가
- 후기에는 고혈압 또는 당뇨 의심 범위가 일부 발생
- 일시적으로 회복되는 날도 포함

### 14.3 demo_recovery

초기에는 비정상 수치가 있으나 꾸준한 운동, 식단 관리, 수면 개선을 통해 건강 수치가 좋아지는 사용자.

- 초기에는 높은 BMI, 혈압, 공복혈당
- 초기 적응기와 정체기를 포함
- 걸음 수와 수면 시간이 점차 증가
- 몸무게와 혈압, 공복혈당이 완만하게 개선
- 후기에는 정상 또는 주의 범위로 이동

### 14.4 현실형 합성 시계열 생성 원칙

가상 데이터는 단순한 선형 증가·감소나 필드별 독립 난수로 만들지 않는다. 각 사용자의 전날 상태와 생활 습관 변화가 다음 날 데이터에 일부 이어지도록 시계열 연속성을 부여한다.

공통 생성 요소는 다음과 같다.

```text
개인별 시작 상태와 1년 장기 추세
+ 전날 수치를 일부 이어받는 연속성
+ 평일·주말 활동 패턴
+ 계절에 따른 활동량·수면의 완만한 변화
+ 수면·활동량과 혈압·혈당 사이의 느슨한 방향성
+ 체중 변화가 다른 지표보다 느리게 반영되는 시차
+ 날짜별 제한된 무작위 변동
+ 드문 피로·과로·휴식·생활 습관 반등 구간
```

세 사용자별 추가 원칙은 다음과 같다.

- `demo_stable`: 정상 범위 안에서 작은 변동을 유지한다. 겨울철 활동량 감소, 간헐적 수면 부족, 휴식일로 인한 일시적 주의 수치를 허용하되 연간 평균은 안정적으로 유지한다.
- `demo_decline`: 초기 경고 단계에서 활동량과 수면이 점진적으로 감소하고 체중이 증가하도록 한다. 중간에 짧은 생활 개선 구간을 두되 이후 재악화되며, 후기에는 고혈압 또는 당뇨 의심 수치가 일부 나타나도록 한다.
- `demo_recovery`: 초기 비정상 상태에서 운동·식단·수면 관리가 단계적으로 강화되도록 한다. 적응기, 감량기, 정체기, 일시적 반등을 포함하며 후기에는 정상 또는 주의 범위로 개선되도록 한다.

현실성 제약은 다음과 같다.

- 하루 사이 체중이 비현실적으로 급격히 변하지 않도록 제한한다.
- 수축기 혈압은 항상 이완기 혈압보다 높아야 한다.
- 걸음 수, 수면 시간, 혈압, 공복혈당은 사전에 정한 허용 범위를 벗어나지 않아야 한다.
- 수면 부족이나 활동량 감소가 혈압·혈당 상승을 반드시 결정하지는 않으며, 확률적이고 느슨한 관계로만 반영한다.
- 생활 습관의 영향은 당일에 즉시 모두 반영하지 않고 일부 시차를 둔다.
- 특정 날짜의 일시적 악화나 회복이 전체 장기 추세를 깨뜨리지 않도록 한다.

랜덤 시드를 고정하여 같은 스크립트를 실행하면 동일한 결과가 생성되도록 한다.

스크립트를 재실행해도 같은 사용자·같은 날짜의 기록이 중복 생성되지 않아야 한다.

본 데이터는 건강 지표 간의 일반적인 방향성과 장기 변동을 참고하여 설계한 시연용 합성 시계열이다. 실제 사람의 건강 변화를 예측하거나 생활 습관과 질환 사이의 의학적 인과관계를 입증하는 모델이 아니다.

## 15. 비기능 요구사항

- 서버를 재시작해도 사용자와 건강 기록이 유지되어야 한다.
- 비밀번호 평문을 파일이나 DB에 저장하지 않아야 한다.
- 다른 사용자의 기록에 접근할 수 없어야 한다.
- SQLite 외래키 제약을 활성화해야 한다.
- 데이터 변경 쿼리는 트랜잭션으로 처리해야 한다.
- API 오류는 적절한 HTTP 상태 코드와 메시지를 반환해야 한다.
- Swagger 문서에서 요청·응답 구조를 확인할 수 있어야 한다.
- Docker 컨테이너에서 실행 가능해야 한다.
- SQLite 파일은 Git 저장소와 Docker 이미지에 포함하지 않아야 한다.
- Docker 볼륨을 사용하면 컨테이너를 재생성해도 DB가 유지되어야 한다.

- 단건 조회

### 16.2 1년치 데이터 검증

- 각 사용자별 기록 수 365건
- 전체 기록 수 1,095건
- 사용자별 중복 날짜 없음
- 2026년 1월 검색 결과 31건
- 날짜 오름차순 정렬
- 사용자별 검색 결과 분리
- 사용자별 통계 계산 정상
- 모든 기록에서 수축기 혈압이 이완기 혈압보다 높음
- 체중·걸음 수·수면·혈압·공복혈당이 정한 허용 범위 안에 있음
- 하루 사이 체중 변화가 정한 최대 변동폭을 넘지 않음
- `demo_stable`의 첫 30일과 마지막 30일 평균이 큰 차이 없이 유지됨
- `demo_decline`의 마지막 30일 평균에서 체중·혈압·혈당이 상승하고 걸음 수·수면이 감소함
- `demo_recovery`의 마지막 30일 평균에서 체중·혈압·혈당이 하락하고 걸음 수·수면이 증가함
- 악화와 회복 과정에 일시적 반등·정체 구간이 존재하되 전체 장기 추세와 일치함
